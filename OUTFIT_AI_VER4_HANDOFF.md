本資料は、現在公開中のOUTFIT AI Ver4 第1フェーズを次のChatGPT/Codexセッションへ正確に引き継ぎ、次の改善仕様を検討するための基準資料である。

## 2. プロダクトの現在地

OUTFIT AIは、所有するシャツ・パンツ・靴を登録し、予定・天気・服の色などから、その日に履く靴をルールベースで提案する個人向けPWAである。

現在の中心体験は次のとおり。

1. マスターへ服・靴・靴箱を登録する
2. 今日の予定、シャツ、パンツを選ぶ
3. SVGキャラクターで色合わせを確認する
4. おすすめの靴3候補から一足を選ぶ
5. 着用履歴を保存する
6. 靴カルテで使用状況や手入れを確認する
7. 気づきから長期未使用、メンテナンス、情報不足を把握する

プロダクトコンセプトは、単なる管理台帳ではなく「毎日開きたくなる靴スタイリスト」である。

- GitHub Pagesで公開できる静的PWAを維持する

- ブランドサイトの画像を利用しない
- アイテム画像はユーザー撮影・選択画像のみ利用する
- 既存の端末内データを壊さない
- 大規模リファクタリングは避け、段階的に改善する

## 6. マスター登録体験

登録はモーダル内で完結し、途中で別画面へ遷移しない。

現在の流れ:

1. 写真撮影またはライブラリから選択
2. 大きな写真プレビューを表示
3. Canvasで写真を圧縮
4. 平均RGBによる簡易色解析
5. 色と種類を使った仮名称を設定
6. 名称候補を表示
7. ユーザーが情報を修正
8. 保存

登録画面には「1 写真 / 2 解析 / 3 名称 / 4 保存」のステップ表示がある。

### 登録可能情報

共通:

- 写真
- 名称
- ブランド
- 色
- 種類
- フォーマル度
- 自由タグ
- お気に入り
- 商品名
- 型番
- 購入店
- 購入日
- 価格
- 商品URL
- メモ
- タグ写真登録済みフラグ

種別固有:

- シャツ: 袖、柄
- パンツ: 丈
- 靴: 素材、雨対応、歩きやすさ、脱ぎやすさ、靴箱写真登録済みフラグ

### 写真処理

- 最大辺1000pxへ縮小
- JPEG品質0.76
- Base64 Data URLへ変換
- `localStorage`内のアイテムに直接保存

### 色解析

画像を24×24pxへ縮小して平均RGBを計算し、次の色へ分類する。

- 白
- 黒
- グレー
- ネイビー
- 青
- ベージュ
- ブラウン
- グリーン
- 赤
- その他

これは画像認識AIではなく、単純な平均色ルールである。背景色や写真の占有率に強く影響される。

- 生成AIは使用していない

- 英語表現や種類ごとの語彙はまだ少ない

## 8. 「情報を育てる」機能

登録後の編集画面から次の写真を追加できる。

- タグ写真
- 靴箱写真（靴のみ）

追加時にはブランド、種類、タグを「追加候補」として表示し、チェックされた候補だけ保存時に反映する。自動更新はしない。

ただし現状は重要な簡易実装である。

- タグ写真や靴箱写真の画像内容は解析していない
- 写真本体も専用フィールドには保存していない
- `tagPhoto` / `shoeBoxPhoto`の真偽値だけを保存する
- ブランド候補等は既存入力値または固定的な仮候補である

次回仕様でOCRや本物のルール解析を求める場合、生成AIなしでどこまで実現するかを先に定義する必要がある。

## 9. 箱だけ登録

靴が手元にない状態で靴箱だけを登録できる。

靴箱データ:

```js
{
  id,
  image,
  brand,
  productName,
  modelNumber,
  color,
  memo,
  linkedShoeId,
  createdAt

```

特徴:

- `linkedShoeId`が空なら「未関連」
- 自動関連付けは行わない
- ユーザーが選択した靴だけに関連付ける
- 後から関連先を変更可能
- 未関連箱は今日画面の気づき対象
- 関連済み箱は靴カルテの情報充実度へ反映

## 10. 情報充実度

カルテで0〜100%を表示する。

現在の評価対象:

- 写真
- ブランド
- 名称
- 種類
- 色
- タグ
- 商品名
- 型番
- 購入店
- 購入日
- メモ
- タグ写真登録済み
- 靴の場合は靴箱写真登録済み、または関連済みの箱

すべてをほぼ同じ重みで判定している。仕様例の72%・85%・95%へ厳密に一致させる重み付けではない。

## 11. 推薦ロジック

推薦は各靴を初期50点として加減点し、上位3件を返す。

### 現在のルール

| 条件 | 点数 |
|---|---:|
| 仕事・営業・フォーマルで靴のフォーマル度4以上 | +22 |
| 同予定でフォーマル度不足 | -10 |
| 休日でフォーマル度3以下 | +12 |
| 長時間歩く | `walk × 6` |
| 雨条件で雨対応`good` | +25 |
| 雨条件で雨対応`bad` | -30 |
| 雨条件でスエード | -25 |
| 白シャツ＋黒靴 | +10 |
| ネイビーパンツ＋黒またはブラウン靴 | +10 |
| グレーパンツ＋黒靴 | +10 |
| お気に入り | +8 |
| 使用回数0 | +12 |
| 当日使用済み | -25 |

最終スコアは0〜100へ丸める。理由は加減点した条件文を「・」で連結する。

### 星表示

`Math.round(score / 20)`を1〜5へ丸める。そのため92点と100点はいずれも星5個になる。

### 現在考慮していない要素

- 気温と季節
- シャツ・パンツの種類同士の相性
- 全般的な配色理論
- 前回使用からの適切な休ませ期間
- 革靴の連続使用回避
- メンテナンス状態による減点
- ユーザーが候補を選ばなかった履歴
- 実際の着用満足度
- 旅行の日数
- 場所、屋内外、ドレスコード詳細

- 履歴を先頭へ追加

## 22. 既知の課題とリスク

### 22.1 データ整合性

履歴がアイテムIDではなく名称を保存している。アイテム名を変更すると、過去履歴と56日カレンダーの関連が切れる可能性がある。同名アイテムも区別できない。

推奨対応:

- 履歴へ`shirtId`、`pantsId`、`shoesId`を追加
- 名称は記録時スナップショットとして併記
- 既存履歴を壊さない段階的移行処理を用意

### 22.2 保存容量

Base64画像を`localStorage`へ保存するため、登録数が増えると容量上限へ達しやすい。Base64化による容量増加もある。

推奨対応:

- 画像をIndexedDBへ移行
- メタデータだけをlocalStorageまたはIndexedDBへ保存
- 画像容量・枚数を画面へ表示
- 移行時は既存Base64画像を段階的に移す

### 22.3 バックアップ

JSONエクスポート、インポート、端末移行がない。ブラウザデータ削除で全データを失う。

### 22.4 着用操作

- 誤操作の取り消し不可
- 履歴の編集・削除不可
- 同日重複を防止しない
- 使用回数と履歴の再計算機能なし

### 22.5 情報追加写真

タグ・靴箱写真の実画像を保持せず、内容も解析しない。UI上の期待と実処理に差がある。

### 22.6 推薦精度

- 保存JSONの厳密なスキーマ検証がない

### 22.8 保守性

- 自動テストなし
- lint・formatなし
- READMEや正式なデータ移行仕様なし

### 22.9 アクセシビリティ

## 27. 引き継ぎ時の重要な結論
